Promise知识点梳理

之前对promise进行过一定的了解，但是仍然对其中部分内容理解不够全面，今天就针对Promise进行一下小小的总结。

Promise的出现

ECMAScript6及之后的几个版本加大了对异步编程机制的支持，而Promise引用类型就是ECMAScript6因此新增的类型，使得优雅地定义和组织异步逻辑成为现实，而且在此之后地几个版本更是增加了使用async和await关键字定义异步函数的机制。

异步编程

JavaScript是一种单线程事件循环模型，同步操作和异步操作是代码所要依赖的核心机制，异步行为是为了优化因计算量大而时间长的操作（其实当我们不想因为某个异步操作而阻塞线程执行时，都可以使用异步编程）

同步和异步

  • 同步行为对应内存中顺序执行的处理器指令，每条指令都会严格按照它们出现的顺序来执行，每条指令执行后也能立即存储在系统本地（例如寄存器或者系统内存）
  • 异步行为类似于系统中断，即当前进程外部的实体可以触发代码执行，异步操作经常是必要的，因为强制进程等待一个长时间的操作是不可行的（同步操作必须要等），如果代码要访问一些高延迟的资源，例如向远程服务器发送请求并等待响应，那么就会出现长时间的等待。另外，到底什么时候会触发这个中断，这对JavaScript运行时来说是一个黑盒，在排定回调之后基本无法知道系统状态何时变化。设计一个能够在异步行为完成后通知其他代码的系统是非常难的，JavaScript在实现这样一个系统的过程中也经历了几次迭代。

Promise前的异步编程模式

在早期的JavaScript中，支支持定义回调函数来表明异步操作完成，串联多个异步操作是一个常见的问题，通常需要深度嵌套的回调函数（即“回调地狱”）来解决 

function double(value,success,failure){
setTimeout(()=>{
    try{
        if(typeof value !=='number'){
            throw 'Must provide number as first argument'
        }
        success(2*value)
    }catch(e){
        failure(e)
    }
},1000)
}
const successCallback = (x)=>{
double(x,(y)=>console.log(`Success:${y}`))
}
const faliureCallback = (e) => {
console.log(`Failure:${e}`)
}
double(3,successCallback,failureCallback)

以上为回调嵌套的一个实例，大约2000ms之后输出Success:12
可以看出上述代码看上去还是比较复杂的

Promise

于是就出现了Promise,翻译成中文是“期约”，这是对尚不存在结果的一个替身。这个名字在1976年就被提出，到了2010年，CommonJS项目实现的Promises/A规范日益流行，2012年Promises/A+组织fork了CommonJS的Promises/A建议，并以相同的名字制定了Promises/A+规范，成为了ECMAScript6规范实现的范本。

Promise基础

Promise状态

pending -> fulfilled
pending -> rejected
而且Promise的状态是私有的，不能直接通过JavaScript检测到，也不能被外部JavaScript代码修改，这样可以避免根据读取到的Promise状态，以同步方式处理Promise对象

new Promise()的过程

这块跟手写Promise有点关系，理解了使用Promise之后整个的工作流程对于手写Promise比较有帮助
下面看一个(伪)代码实例

  1. 在创建promise实例时，传入的参数为一个执行器函数。这个函数有两个参数，分别是resolve和reject，这两个函数用来改变promise的状态，而且promise的状态只能由pending->fulfilled/rejected改变一次，改变之后就不能再改变；在执行器函数中可以完成异步任务，并根据异步任务的返回结果调用resolve()或者reject()
  2. Promise.prototype中定义了.then() .catch() .finally()等方法，作用都是类似的，指定了Promise实例的成功或失败回调，当Promise实例的状态发生改变时就会调用对应的回调函数

.then()返回的结果为Promise实例

.catch()及错误处理

同步代码抛错误用throw（不仅是可以手动抛错误，如果我们写的代码里边有错误，如使用了未定义变量，跟手动抛的效果是一样的） ，而且可以用try catch捕获，异步代码抛错误用reject()
在new Promise()的执行器函数内部，不管是同步抛出的错误还是异步reject()抛出的错误都可以在.catch()中或者.then()的失败处理函数中被处理

Promise中resolve()与.then()的关系

这块内容我之前一直有一些琢磨不透，理解这两个函数对于我们理解Promise实现整个异步过程非常有帮助，在查了一些资料之后，我认为应该是以下的流程，这样我也就理解了为什么在Promise状态改变前后在.then()中注册的回调函数都可以被执行，另外就是rejected()和resolve()是完全一样的，反过来去理解就行

  1. resolve()方法
  • 如果Promise是pending，则将状态改为fulfilled
  • 将Promise的value赋值为resolve参数
  • 如果有onFulfilledCallback,则将其推到微队列中
  1. then()方法
  • 如果Promise是pending状态：
    (1) 将then的onFulfilled方法追加保存到Promise的onFulfilledCallback，即规范中的PromiseFulfilledReactions列表末尾；
    (2) 将then的onejected方法追加保存到Promise的onRejectedCallback，即规范中的PromiseRejectedReactions列表末尾；
  • 如果Promise的状态是fulfilled状态，则把onFulfilled推到微任务队列
  • 如果Promise的状态是rejected状态，则把onRejected推到微任务队列中

这里再提醒一点，其实new Promise()和.then()都是同步的，只不过在实例化Promise对象的过程中使用的执行器函数中是可以放异步代码的，然后通过上述机制实现这个异步效果

下面看一个关于这个知识点的代码 

// 1. 把定时器任务推到宏队列中
// 13.执行宏队列，打印 0
setTimeout(() => {
  console.log(0)
}, 0);
// 2. 调用 resolve，此时 p1 的状态变为 fulfilled
let p1 = new Promise((resolve, reject) => {
  resolve({name: 1})
})
// 3. 执行 then，把 onFulfilled 推到微队列
// 9. 执行微队列，打印 1
// 9. p2 状态变为 fulfilled, 将 p2 的 onFulfilledCallback 推到微队列
let p2 = p1.then(
  () => {
    console.log(1)
  }
)
// 4. 执行 then，p2 是 pending 状态，所以保存 onFulfulled 到 P2 的 onFulfilledCallback 中
// 11. 执行微队列，打印 2
let p3 = p2.then(value => {
  console.log(2)
})
// 5. 调用 resolve，此时 p4 的状态变为 fulfilled
let p4 = new Promise((resolve, reject) => {
    resolve('Promise')
})
// 6. 执行 then，把 onFulfilled 推到微队列
// 10. 执行微队列，打印 3
// 10. p5 状态变为 fulfilled, 将 p5 的 onFulfilledCallback 推到微队列
let p5 = p4.then(
  ()=> {
    console.log(3)
  },
  error => console.log(error)
)
// 7. 执行 then，p5 是 pending 状态，所以保存 onFulfulled 到 p5 的 onFulfilledCallback 中
// 12. 执行微队列，打印 4
let p6 = p5.then(value => {
  console.log(4)
})
// 8. 输出 5
console.log(5)

结果是5，1，3，2，4，0

Promise涉及的宏任务和微任务的关系

异步任务多种多样，在处理异步任务的过程中，存在宏任务队列和微任务队列两种队列：

  • 宏任务队列：setTimeout setInterval requrestAnimationFrame
  • 微任务队列：new promise().then(回调) process.nextTick，优先级更高

接下来再看一个例子(结合.resolve()和.then()的关系食用) 

    let p = new Promise((resolve,reject)=>{
        setTimeout(()=>{
            console.log('setTimeout!')
            resolve()//宏任务不执行，微任务就没法创建，即使这里边的console.log()和resolve()这两行代码互换位置也不会影响打印结果
        },0)
        console,log('promise!')
    }).then(value=>console.log('成功！'))//'成功！'最后打印，因为在Promise的构造函数中的执行器函数里边的setTimeout虽然是宏任务，但是在执行完这个setTimeout之前，还没有执行resolve()，也就是p的状态还没有转换，成功回调还没有推到微任务队列
    console.log('hhhhhh')

上述代码的打印结果应该是:
promise!
hhhhhh
setTimeout!
成功！

async和await

async其实是promise的语法糖，使得异步任务的执行看上去像是在执行同步任务，function前边加async就相当于是new Promise()的过程，（当然在这个funcion里边显式地new Promise()并作为返回值也是可以的），对这个函数同样可以使用.then()方法，而且在这个async function中的错误也可以通过.then()或者.catch()捕获，或者直接在函数中用try catch包起来捕获
而针对.then()方法的语法糖就是await，可以在await后边直接返回内容，避免了.then()回调的形式（而且await返回的是这个新的promise里边resolve时的参数），遇到await时，要等到它后边的promise状态改变之后再往下走
如果是立即执行函数的话，使用await的外层函数也要记得加async

class和await的结合

如果一个类中有then()方法，则可以被包装为promise，这个then()方法感觉比较类似new Promise()里边的执行器函数

生成器

生成器使得js的执行真正拥有异步的能力了，也就是任务执行过程中可以被打断，在被打断之前将当前任务的状态保存起来，先去执行一会别的，过一会再来恢复当前任务
生成器在js中是以特殊的函数形式出现的，需要在function和函数名之间加一个*，用yield关键字让出函数执行的权限，调用的时候就是像普通函数一样正常调用，但是调用之后并不执行里边的代码，而是返回一个类似于控制器（迭代器）的东西，对这个迭代器调用.next()时往下走

生成器可以与外界交互/通信

  • yield后的值可以被接收，就像return语句一样
  • 除了可以返回一些值，还可以传入一些值（实现双向通信了)
    “取值的时候用上一个next()，传值的时候用下一个next() “ 
 function *main() {
        console.log('First!')
        yield "Hello World" //yield后边的内容可以在it.next()被接到
        console.log('Second!')
        yield
        var i = yield "Third!" //当调用it.next()并传参时，传递的参数会代替yield的内容
    }
    var it = main()
    let ret1 = it.next() // First!
    console.log(ret1) // {value:'Hello World',done:false}
    console.log('hhh')
    let ret2 = it.next() // Second!
    console.log(ret2) //{value:undefined,done:true}
    //最后一个value不是一定是undefined，可以由return返回

生成器和异步任务的关系

生成器在设计之初并不是用来处理异步任务的，而是用来生成数据的，但生成器可以把任务打断让其他任务进来的特性非常适合处理异步任务，因此很多语言都有生成器，并用来完成异步任务

错误的处理

可以用it.throw()抛出异常（抛到yield所在处，并把返回值的done设置成true）并用try catch处理（再次强调，同步代码才能用try catch捕获，异步代码不行）
